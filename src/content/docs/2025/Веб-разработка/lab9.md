## Практика

Продолжите работу на ветке из прошлого занятия и допишите новый функционал.

Ознакомьтесь с [соглашением о коммитах](https://www.conventionalcommits.org/ru/v1.0.0/). Как подспорье можно использовать [Conventional Commits без лишних слов: ваша шпаргалка](https://habr.com/ru/articles/867012/).

В рамках выполнения этой работы делайте отдельные небольшие коммиты с понятными сообщениями по соглашению.

### Создание БД

Хоть в node.js v22.5.0 добавили встроенный модуль для работы с SQLite, он пока нестабилен, поэтому используйте пакет sqlite3 для интеграции с SQLite.

Подумайте какие таблицы могут понадобится для работы вашего приложения. Создайте файл example.sql, в который поместите команды для создания таблиц и их наполнения.

Используя утилиту sqlite3, создайте файл базы данных SQLite с именем comments.db в каталоге проекта. Этот файл стоит добавить в .gitignore.

Используя функцию .read() в утилите sqlite3 наполнить БД тестовыми данными из example.sql.

### Связь БД и приложения

Создайте новый репозиторий, чтобы в приложении заменить работу с json на SQLite. В идеале эти изменения должны затронуть только слой работы с данными и его импорты в другие файлы, остальное должно остаться неизменным.

При получении списка всех сохраненных моделей сервер должен уметь обрабатывать строки запросов URL, например `/models?page=2&limit=5` и через это реализовать пагинацию.

Пагинация обычно включает в себя два ключевых параметра номер страницы (page): указывает конкретную страницу данных, которую необходимо получить. Каждая страница содержит подмножество всего набора данных. И количество на страницу (limit): указывает максимальное количество элементов для отображения на каждой странице.

Разберите входящий HTTP-запрос, чтобы извлечь параметры пагинации. В node.js для этой задачи есть модуль querystring. Используйте эти параметры в сервисе для запроса к базе данных SQLite и извлечения соответствующего подмножества моделей. Отправьте заданное количество моделей клиенту в качестве ответа в формате JSON. Пример ответа ниже.

Модуль sqlite3 использует функции обратного вызова, для перехода на объекты promise можно использовать из модуля util функцию promisify.

Обязательно обработайте ошибки, которые могут возникнуть при работе с БД.

```json
{
    "page": 1,
    "limit": 10,
    "totalPages": 3,
    "models": [
        {"id": 1, ...},
        {"id": 2, ...},
        ...
    ]
}
```

где,

- "page": Номер текущей страницы.
- "limit": Максимальное количество моделей на странице.
- "totalPages": Общее количество страниц с учетом имеющихся моделей и указанного ограничения.
- "comments": Массив, содержащий модели на странице, каждая из которых представлена в виде объекта.

В пользовательский интерфейс приложения добавьте необходимые ссылки для возможности использовать пагинацию.

### Дополнение по удалении моделей

Проверьте что при удалении модели DELETE-запрос удаляет модель из БД, в теле ответа ничего не возвращается, только статус 204 (No Content).

### Обработка ошибок

Обработайте возможные ошибки:

- Если пришел некорректный запрос, возвращайте ошибку 400-й группы.
- Если ресурс не найден, возвращайте 404 (Not Found).
- Если ошибка возникла на стороне сервера, отправляйте ответ с кодом 500-й группы.

### Очистка кода

Помните, что основное назначение вашего приложения это демонстрация и работа с 3D-моделями пользователей. Теперь ваша задача из приложения удалить лишние части кода. Удалять необходимо по одной части за раз, работоспособность остального проекта не должна быть задета и именно этому изменению должен соответствовать коммит.

### Тестирование

Проведите ручное тестирование, убедитесь что ничего не сломалось и не изменило поведение с прошлых работ. Также проведите нагрузочное тестирование, используя код из прошлой работы, чтобы убедиться что нет радикальной деградации скорости работы сервера.
