**Цель**: **Цель**: изучить основы работы с Node и http модулем.

## Подготовка

Повторите концепцию функций обратного вызова. Изучите события и их обработку в node.js. Изучите класс EventEmitter, так как http.Server наследуется от него. Посмотрите на добавление обработчиков событий и события, что может испускать сервер и объект запроса (наследуется от stream.Readable), у которого есть события data, end, error. Просмотрите документацию модуля node:http.

В документации Node.js примеры включайте с ESM.

## Выполнение

Пока с клиентской частью мы закончили, теперь надо разработать серверную часть, чтобы потом объединить обе составляющие. Поэтому вернитесь на ветку main и создайте и переключитесь на ветку backend. Если остались какие-то файлы от работы на ветке frontend, то удалите их.

Инициализируйте проект как в прошлой работе `npm init -y`. В файле package.json первым свойством поставьте `"private": true` и удалите все необязательные (опциональные) свойства. Укажите что проект работает с ES-модулями в свойстве type.

Установите [eslint](https://eslint.org/) командой `npm init @eslint/config@latest`, но теперь для проверки кода для node.js. Подсказки как отвечать на вопросы при установке:

1. What do you want to lint? - **javascript**
1. How would you like to use ESLint? - **To check syntax and find problems**
1. What type of modules does your project use? - **JavaScript modules (import/export)**
1. Which framework does your project use? - **None of these**
1. Does your project use TypeScript? - **No**
1. Where does your code run? - **Node**
1. Would you like to install them now? - **yes**
1. Which package manager do you want to use? - **npm**

В файл `eslint.config.mjs` добавьте правила из прошлой работы. Добавьте в проект файл `.gitignore` и исключите из коммитов папку `node_modules/`. Сделайте коммит с сообщением об установке зависимостей.

Далее используя возможности git создадим параллельно папку для ветки frontend. `git worktree add ../frontend frontend`. Путь `../frontend` в команде можете заменить на свой вариант. Командой `git worktree list` можно проверить что репозиторий в курсе о новой папке. Это возможность git сейчас часто используется вместе с агентными системами написания кода, кода их можно запускать параллельно решать несколько задач в проекте. Теперь в редакторе кода можете открыть папку с клиентской частью, установить зависимости `npm ci` и проверить работоспособность прошлой работы.

В package.json в script добавьте команду с именем `dev`, которая будет запускать `node --watch ./main.js`. Имя файла у вас может отличаться. Научитесь запускать проект с ее помощью, чтобы не перезапускать сервер вручную в процессе работы. Также добавить команду "serve" для запуска основного файла в качестве сервиса.

В коде создайте экземпляр сервера http и добавьте обработчик входящих запросов (событие 'request'). В этом обработчике, используя информацию из запроса request о пути (request.path), методе (request.method) и данные тела запроса по событиям data и end, восстановите работоспособность вашего приложения, заменив json-server своим сервером. Для проверки работоспособности возможно придется изменить адрес и порт сервера на стороне клиента. Для работы с путем и хостом может быть полезен модуль [URL](https://nodejs.org/docs/latest/api/url.html).

Для работы с файлом db.json используйте модуль node:fs/promises, так как работа с promise вам уже знакома. При старте сервера, вы можете считать данные из файла и преобразовать их в объект js. При get запросах вы можете забирать данные сразу из объекта в памяти. Если входящий запрос должен изменить информацию в объекте, то вы добавляете новую информацию к объекту, и записываете в файл, если возникла ошибка записи, то об этом сообщается в ответе сервера клиенту.

При запросах на несуществующие пути или при неверных данных в теле запроса, необходимо возвращать 400 Bad Request.

Также в консоль выводится сообщения:

- о начале работы - ip и номер порта,
- при открытии нового подключения к нему,
- при получении любого запроса с информацией о запросе.

## Требования к коду и результат выполнения

- Код отформатирован, сохранен в системе контроля версий.
- Отсутствуют ошибки от ESLint.
- Отсутствуют ошибки в консоли.
- Код разбит на логические модули.
